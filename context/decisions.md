### 2026-07-03 — status.json sync mechanism, language, and slug convention

- **Made by:** Coder
- **Context:** The plan (`context/plan.md`) and Manager's task breakdown (`context/tasks.md`) called for `status.json` — a machine-readable mirror of the phase checked in `context/project.md` — to stay in sync automatically. Options considered per the plan: manual convention (reject — relies on discipline, the exact drift this feature prevents), CI-based regeneration on push (reject — no confirmed hosted CI for this repo, adds latency), and a git pre-commit hook (accepted). Language options were Node.js or Python (both present on this machine); Node.js was the plan's default given zero extra dependencies and no existing build tooling in `agent-brain`.
- **Decision:**
  - **Mechanism:** a generator script (`scripts/update-status.js`) plus a git pre-commit hook (`scripts/git-hooks/pre-commit`) that runs it and stages the result. The hook's source is tracked in-repo (not just `.git/hooks/`, which isn't version-controlled) and requires a one-time local `git config core.hooksPath scripts/git-hooks` — documented in `scripts/git-hooks/README.md`. This has been run in the current working directory, so the hook is live for future commits made here.
  - **Language:** Node.js, built-ins only (`fs`, `path`) — no `package.json`, no dependencies added.
  - **Slug convention:** lowercase-hyphenated, derived directly from the six bold phase labels in `context/project.md`: `planning`, `delegated`, `in-progress`, `in-review`, `awaiting-approval`, `shipped`. The generator hardcodes these six labels (in this exact order and wording) as its parse target; if the checklist wording changes, the script must change with it.
  - **File tracking:** `status.json` is tracked in the repo (not gitignored) and is fully derived — it is never hand-edited, only ever regenerated by the script.
  - **Error handling:** the generator fails loudly (non-zero exit, message on stderr) and leaves `status.json` untouched if zero or 2+ of the six phase boxes are checked, or if `context/project.md` can't be read or parsed. Verified against all 6 valid states and both 0-checked and 2-checked invalid states; `status.json` byte-for-byte unchanged (md5-verified) across the failing runs.
- **Impact:** Any future agent reading or extending `status.json` should treat it as generated output, not a source of truth to hand-edit, and should run `node scripts/update-status.js` (or rely on the pre-commit hook) rather than writing to it directly. Anyone cloning or setting up a fresh working directory for this repo needs to run `git config core.hooksPath scripts/git-hooks` once for the hook to take effect there, since `.git/config` isn't shared across working directories. Note: the end-to-end "hook runs during a real `git commit`" test was verified by invoking `scripts/git-hooks/pre-commit` directly rather than via an actual commit, because this build was explicitly instructed not to create a commit — a human or the Reviewer should do one real test commit to fully close that verification step out.

### 2026-07-03 — status.json feature reviewed and shipped

Verdict: SEND BACK on that single item, with the note that the human/orchestrator closing it was acceptable given very low risk.
- **Decision:** Close the gap with the shipping commit rather than a throwaway test commit. The phase was advanced In review → Shipped in `context/project.md`, and the commit that ships the feature exercises the hook end-to-end — the hook regenerates `status.json` to `shipped` and stages it into that same commit, satisfying plan.md §5 item 3 as a live test rather than an isolated one.
- **Impact:** Feature is accepted and shipped. The three tasks are in Done; the board phase is Shipped. Future phase changes to `context/project.md` will keep `status.json` current automatically on each commit via the now-verified hook.
